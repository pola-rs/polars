1//! # Polars: *<small>DataFrames in Rust</small>*
2//!
3//! Polars is a DataFrame library for Rust. It is based on [Apache Arrow](https://arrow.apache.org/)'s memory model.
4//! Apache Arrow provides very cache efficient columnar data structures and is becoming the defacto
5//! standard for columnar data.
6//!
7//! ## Quickstart
8//! We recommend building queries directly with [polars-lazy]. This allows you to combine
9//! expressions into powerful aggregations and column selections. All expressions are evaluated
10//! in parallel and queries are optimized just in time.
11//!
12//! [polars-lazy]: polars_lazy
13//!
14//! ```no_run
15//! use polars::prelude::*;
16//! # fn example() -> PolarsResult<()> {
17//!
18//! let lf1 = LazyFrame::scan_parquet("myfile_1.parquet", Default::default())?
19//!     .group_by([col("ham")])
20//!     .agg([
21//!         // expressions can be combined into powerful aggregations
22//!         col("foo")
23//!             .sort_by([col("ham").rank(Default::default(), None)], SortMultipleOptions::default())
24//!             .last()
25//!             .alias("last_foo_ranked_by_ham"),
26//!         // every expression runs in parallel
27//!         col("foo").cum_min(false).alias("cumulative_min_per_group"),
28//!         // every expression runs in parallel
29//!         col("foo").reverse().implode().alias("reverse_group"),
30//!     ]);
31//!
32//! let lf2 = LazyFrame::scan_parquet("myfile_2.parquet", Default::default())?
33//!     .select([col("ham"), col("spam")]);
34//!
35//! let df = lf1
36//!     .join(lf2, [col("reverse")], [col("foo")], JoinArgs::new(JoinType::Left))
37//!     // now we finally materialize the result.
38//!     .collect()?;
39//! # Ok(())
40//! # }
41//! ```
42//!
43//! This means that Polars data structures can be shared zero copy with processes in many different
44//! languages.

68//! ## Data Structures
69//! The base data structures provided by polars are [`DataFrame`], [`Series`], and [`ChunkedArray<T>`].
70//! We will provide a short, top-down view of these data structures.
71//!
72//! [`DataFrame`]: crate::frame::DataFrame
73//! [`Series`]: crate::series::Series
74//! [`ChunkedArray<T>`]: crate::chunked_array::ChunkedArray
75//!
76//! ### DataFrame
77//! A [`DataFrame`] is a two-dimensional data structure backed by a [`Series`] and can be
78//! seen as an abstraction on [`Vec<Series>`]. Operations that can be executed on a [`DataFrame`] are
79//! similar to what is done in a `SQL` like query. You can `GROUP`, `JOIN`, `PIVOT` etc.
80//!
81//! [`Vec<Series>`]: std::vec::Vec
82//!
83//! ### Series
84//! [`Series`] are the type-agnostic columnar data representation of Polars. The [`Series`] struct and
85//! [`SeriesTrait`] trait provide many operations out of the box. Most type-agnostic operations are provided
86//! by [`Series`]. Type-aware operations require downcasting to the typed data structure that is wrapped
87//! by the [`Series`]. The underlying typed data structure is a [`ChunkedArray<T>`].
88//!
89//! [`SeriesTrait`]: crate::series::SeriesTrait
90//!
91//! ### ChunkedArray
92//! [`ChunkedArray<T>`] are wrappers around an arrow array, that can contain multiples chunks, e.g.
93//! [`Vec<dyn ArrowArray>`]. These are the root data structures of Polars, and implement many operations.
94//! Most operations are implemented by traits defined in [chunked_array::ops],
95//! or on the [`ChunkedArray`] struct.
96//!
97//! [`ChunkedArray`]: crate::chunked_array::ChunkedArray
98//!
99//! ## SIMD
100//! Polars / Arrow uses packed_simd to speed up kernels with SIMD operations. SIMD is an optional
101//! `feature = "nightly"`, and requires a nightly compiler. If you don't need SIMD, **Polars runs on stable!**
102//!
103//! ## API
104//! Polars supports an eager and a lazy API. The eager API directly yields results, but is overall
105//! more verbose and less capable of building elegant composite queries. We recommend to use the Lazy API
106//! whenever you can.
107//!
108//! As neither API is async they should be wrapped in _spawn_blocking_ when used in an async context
109//! to avoid blocking the async thread pool of the runtime.
110//!
111//! ## Expressions
112//! Polars has a powerful concept called expressions.
113//! Polars expressions can be used in various contexts and are a functional mapping of
114//! `Fn(Series) -> Series`, meaning that they have [`Series`] as input and [`Series`] as output.
115//! By looking at this functional definition, we can see that the output of an [`Expr`] also can serve
116//! as the input of an [`Expr`].
117//!
118//! [`Expr`]: polars_lazy::dsl::Expr
119//!
120//! That may sound a bit strange, so lets give an example. The following is an expression:
121//!
122//! `col("foo").sort().head(2)`
123//!
124//! The snippet above says select column `"foo"` then sort this column and then take the first 2 values
125//! of the sorted output.
126//! The power of expressions is that every expression produces a new expression and that they can
127//! be piped together.
128//! You can run an expression by passing them on one of polars execution contexts.
129//! Here we run two expressions in the **select** context:
130//!
131//! ```no_run
132//! # use polars::prelude::*;
133//! # fn example() -> PolarsResult<()> {
134//! # let df = DataFrame::default();
135//!   df.lazy()
136//!    .select([
137//!        col("foo").sort(Default::default()).head(None),
138//!        col("bar").filter(col("foo").eq(lit(1))).sum(),
139//!    ])
140//!    .collect()?;
141//! # Ok(())
142//! # }
143//! ```
144//! All expressions are run in parallel, meaning that separate polars expressions are embarrassingly parallel.
145//! (Note that within an expression there may be more parallelization going on).
146//!
147//! Understanding Polars expressions is most important when starting with the Polars library. Read more
148//! about them in the [user guide](https://docs.pola.rs/user-guide/expressions).
149//!

165//! ## Compile times
166//! A DataFrame library typically consists of
167//!
168//! * Tons of features
169//! * A lot of datatypes
170//!
171//! Both of these really put strain on compile times. To keep Polars lean, we make both **opt-in**,
172//! meaning that you only pay the compilation cost if you need it.
173//!
174//! ## Compile times and opt-in features
175//! The opt-in features are (not including dtype features):
176//!
177//! * `lazy` - Lazy API
178//!     - `regex` - Use regexes in [column selection]
179//!     - `dot_diagram` - Create dot diagrams from lazy logical plans.
180//! * `sql` - Pass SQL queries to Polars.
181//! * `random` - Generate arrays with randomly sampled values
182//! * `ndarray`- Convert from [`DataFrame`] to [ndarray](https://docs.rs/ndarray/)
183//! * `temporal` - Conversions between [Chrono](https://docs.rs/chrono/) and Polars for temporal data types
184//! * `timezones` - Activate timezone support.
185//! * `strings` - Extra string utilities for [`StringChunked`]
186//!     - `string_pad` - `zfill`, `ljust`, `rjust`
187//!     - `string_to_integer` - `parse_int`
188//! * `object` - Support for generic ChunkedArrays called [`ObjectChunked<T>`] (generic over `T`).
189//!   These are downcastable from Series through the [Any](https://doc.rust-lang.org/std/any/index.html) trait.
190//! * Performance related:
191//!     - `nightly` - Several nightly only features such as SIMD and specialization.
192//!     - `performant` - more fast paths, slower compile times.
193//!     - `bigidx` - Activate this feature if you expect >> 2^32 rows. This is rarely needed.
194//!       This allows Polars to scale up beyond 2^32 rows by using an index with a `u64` data type.
195//!       Polars will be a bit slower with this feature activated as many data structures
196//!       are less cache efficient.
197//!     - `cse` - Activate common subplan elimination optimization
198//! * IO related:
199//!     - `serde` - Support for [serde](https://crates.io/crates/serde) serialization and deserialization.
200//!       Can be used for JSON and more serde supported serialization formats.
201//!     - `serde-lazy` - Support for [serde](https://crates.io/crates/serde) serialization and deserialization.
202//!       Can be used for JSON and more serde supported serialization formats.
203//!     - `parquet` - Read Apache Parquet format
204//!     - `json` - JSON serialization
205//!     - `ipc` - Arrow's IPC format serialization
206//!     - `decompress` - Automatically infer compression of csvs and decompress them.
207//!       Supported compressions:
208//!          - gzip
209//!          - zlib
210//!          - zstd

286//! ## Compile times and opt-in data types
287//! As mentioned above, Polars [`Series`] are wrappers around
288//! [`ChunkedArray<T>`] without the generic parameter `T`.
289//! To get rid of the generic parameter, all the possible values of `T` are compiled
290//! for [`Series`]. This gets more expensive the more types you want for a [`Series`]. In order to reduce
291//! the compile times, we have decided to default to a minimal set of types and make more [`Series`] types
292//! opt-in.
293//!
294//! Note that if you get strange compile time errors, you probably need to opt-in for that [`Series`] dtype.
295//! The opt-in dtypes are:
296//!
297//! | data type               | feature flag      |
298//! |-------------------------|-------------------|
299//! | Date                    | dtype-date        |
300//! | Datetime                | dtype-datetime    |
301//! | Time                    | dtype-time        |
302//! | Duration                | dtype-duration    |
303//! | Int8                    | dtype-i8          |
304//! | Int16                   | dtype-i16         |
305//! | UInt8                   | dtype-u8          |
306//! | UInt16                  | dtype-u16         |
307//! | Categorical             | dtype-categorical |
308//! | Struct                  | dtype-struct      |
309//!
310//!
311//! Or you can choose one of the preconfigured pre-sets.
312//!
313//! * `dtype-full` - all opt-in dtypes.
314//! * `dtype-slim` - slim preset of opt-in dtypes.
315//!
316//! ## Performance
317//! To get the best performance out of Polars we recommend compiling on a nightly compiler
318//! with the features `simd` and `performant` activated. The activated cpu features also influence
319//! the amount of simd acceleration we can use.
320//!
321//! See the features we activate for our python builds, or if you just run locally and want to
322//! use all available features on your cpu, set `RUSTFLAGS='-C target-cpu=native'`.
323//!
324//! ### Custom allocator
325//! An OLAP query engine does a lot of heap allocations. It is recommended to use a custom
326//! allocator, (we have found this to have up to ~25% runtime influence).
327//! [JeMalloc](https://crates.io/crates/tikv-jemallocator) and
328//! [Mimalloc](https://crates.io/crates/mimalloc) for instance, show a significant
329//! performance gain in runtime as well as memory usage.
330//!
331//! #### Jemalloc Usage
332//! ```ignore
333//! use tikv_jemallocator::Jemalloc;
334//!
335//! #[global_allocator]
336//! static GLOBAL: Jemalloc = Jemalloc;
337//! ```
338//!
339//! #### Cargo.toml
340//! ```toml
341//! [dependencies]
342//! tikv-jemallocator = { version = "*" }
343//! ```
344//!
345//! #### Mimalloc Usage
346//!
347//! ```ignore
348//! use mimalloc::MiMalloc;
349//!
350//! #[global_allocator]
351//! static GLOBAL: MiMalloc = MiMalloc;
352//! ```
353//!
354//! #### Cargo.toml
355//! ```toml
356//! [dependencies]
357//! mimalloc = { version = "*", default-features = false }
358//! ```
359//!
360//! #### Notes
361//! [Benchmarks](https://github.com/pola-rs/polars/pull/3108) have shown that on Linux and macOS JeMalloc
362//! outperforms Mimalloc on all tasks and is therefore the default allocator used for the Python bindings on Unix platforms.
363//!
364//! ## Config with ENV vars
365//!
366//! * `POLARS_FMT_TABLE_FORMATTING` -> define styling of tables using any of the following options (default = UTF8_FULL_CONDENSED). These options are defined by comfy-table which provides examples for each at <https://github.com/Nukesor/comfy-table/blob/main/src/style/presets.rs>
367//!   * `ASCII_FULL`
368//!   * `ASCII_FULL_CONDENSED`
369//!   * `ASCII_NO_BORDERS`
370//!   * `ASCII_BORDERS_ONLY`
371//!   * `ASCII_BORDERS_ONLY_CONDENSED`
372//!   * `ASCII_HORIZONTAL_ONLY`
373//!   * `ASCII_MARKDOWN`
374//!   * `MARKDOWN`
375//!   * `UTF8_FULL`
376//!   * `UTF8_FULL_CONDENSED`
377//!   * `UTF8_NO_BORDERS`
378//!   * `UTF8_BORDERS_ONLY`
379//!   * `UTF8_HORIZONTAL_ONLY`
380//!   * `NOTHING`
381//! * `POLARS_FMT_TABLE_CELL_ALIGNMENT` -> define cell alignment using any of the following options (default = LEFT):
382//!   * `LEFT`
383//!   * `CENTER`
384//!   * `RIGHT`
385//! * `POLARS_FMT_TABLE_DATAFRAME_SHAPE_BELOW` -> print shape information below the table.
386//! * `POLARS_FMT_TABLE_HIDE_COLUMN_NAMES` -> hide table column names.
387//! * `POLARS_FMT_TABLE_HIDE_COLUMN_DATA_TYPES` -> hide data types for columns.
388//! * `POLARS_FMT_TABLE_HIDE_COLUMN_SEPARATOR` -> hide separator that separates column names from rows.
389//! * `POLARS_FMT_TABLE_HIDE_DATAFRAME_SHAPE_INFORMATION"` -> omit table shape information.
390//! * `POLARS_FMT_TABLE_INLINE_COLUMN_DATA_TYPE` -> put column data type on the same line as the column name.
391//! * `POLARS_FMT_TABLE_ROUNDED_CORNERS` -> apply rounded corners to UTF8-styled tables.
392//! * `POLARS_FMT_MAX_COLS` -> maximum number of columns shown when formatting DataFrames.
393//! * `POLARS_FMT_MAX_ROWS` -> maximum number of rows shown when formatting DataFrames, `-1` to show all.
394//! * `POLARS_FMT_STR_LEN` -> maximum number of characters printed per string value.
395//! * `POLARS_TABLE_WIDTH` -> width of the tables used during DataFrame formatting.
396//! * `POLARS_MAX_THREADS` -> maximum number of threads used to initialize thread pool (on startup).
397//! * `POLARS_VERBOSE` -> print logging info to stderr.
398//! * `POLARS_NO_PARTITION` -> polars may choose to partition the group_by operation, based on data
399//!   cardinality. Setting this env var will turn partitioned group_by's off.
400//! * `POLARS_PARTITION_UNIQUE_COUNT` -> at which (estimated) key count a partitioned group_by should run.
401//!   defaults to `1000`, any higher cardinality will run default group_by.
402//! * `POLARS_FORCE_PARTITION` -> force partitioned group_by if the keys and aggregations allow it.
403//! * `POLARS_ALLOW_EXTENSION` -> allows for [`ObjectChunked<T>`] to be used in arrow, opening up possibilities like using
404//!   `T` in complex lazy expressions. However this does require `unsafe` code allow this.
405//! * `POLARS_NO_PARQUET_STATISTICS` -> if set, statistics in parquet files are ignored.
406//! * `POLARS_PANIC_ON_ERR` -> panic instead of returning an Error.
407//! * `POLARS_BACKTRACE_IN_ERR` -> include a Rust backtrace in Error messages.
408//! * `POLARS_NO_CHUNKED_JOIN` -> force rechunk before joins.
